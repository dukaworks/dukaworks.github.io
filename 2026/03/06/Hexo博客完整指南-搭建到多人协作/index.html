Hexo 博客完整指南：从搭建到多人协作

Hexo 是一个快速、简洁且高效的静态博客框架。本文将带你从零开始，搭建一个功能完善的技术博客，并支持多人协作写作。

一、环境准备

1.1 安装 Node.js

Hexo 基于 Node.js，需要先安装：

# 下载安装 Node.js (建议 LTS 版本)
# https://nodejs.org

# 验证安装
node -v  # v18.x.x 或更高
npm -v   # 9.x.x 或更高

1.2 安装 Git

# Windows: https://git-scm.com/download/win
# macOS: brew install git
# Linux: sudo apt-get install git

# 配置 Git
git config --global user.name "Your Name"
git config --global user.email "your_email@example.com"

1.3 安装 Hexo

# 全局安装 Hexo CLI
npm install -g hexo-cli

# 验证安装
hexo -v

二、创建博客项目

2.1 初始化项目

# 创建博客目录
hexo init my-blog
cd my-blog

# 安装依赖
npm install

2.2 项目结构说明

my-blog/
├── _config.yml          # 站点配置文件
├── package.json         # 项目依赖
├── scaffolds/           # 文章模板
├── source/              # 源代码
│   ├── _posts/         # 文章目录
│   ├── _drafts/        # 草稿目录
│   ├── about/          # 关于页面
│   └── images/         # 图片资源
├── themes/             # 主题目录
└── public/             # 生成的静态文件

2.3 基础配置

编辑 _config.yml：

# 站点信息
title: 你的博客名称
subtitle: 博客副标题
description: 博客描述
keywords: 关键词,用逗号分隔
author: 作者名
language: zh-CN
timezone: Asia/Shanghai

# URL
url: https://yourname.github.io
permalink: :year/:month/:day/:title/

# 部署配置
deploy:
  type: git
  repo: https://github.com/yourname/yourname.github.io.git
  branch: main

六、多人协作写作

6.1 协作方案

多人协作 Hexo 博客的推荐方案（双仓库）：

hexo-source 仓库（私有，存放源码）
├── source/_posts/     ← 文章源码
├── themes/            ← 主题文件
├── _config.yml        ← 站点配置
├── _config.next.yml   ← 主题配置
└── package.json       ← 依赖配置
       ↓
GitHub Actions 自动构建部署
       ↓
yourname.github.io 仓库（公开，GitHub Pages）
├── index.html         ← 生成的静态页面
├── 2026/03/06/        ← 文章页面
└── ...                ← CSS/JS/图片资源

为什么用双仓库？

仓库 用途 可见性 内容
hexo-source 源码仓库 私有 Markdown文章、配置、主题
yourname.github.io 部署仓库 公开 生成的HTML静态文件

优点：

  • ✅ 源码私密，草稿不会暴露
  • ✅ 多人协作更安全
  • ✅ 构建历史清晰分离
  • ✅ 支持私有文章/草稿

6.2 仓库设置

1. 创建源码仓库（hexo-source）

# 在 GitHub 创建私有仓库：hexo-source
# 不要初始化 README，保持为空

2. 创建部署仓库（yourname.github.io）

# 创建公开仓库，名称必须是 yourname.github.io
# 用于 GitHub Pages 部署

3. 配置部署密钥

为了让 GitHub Actions 有权限推送到部署仓库，需要配置 Deploy Key：

步骤1：生成 SSH 密钥对

ssh-keygen -t rsa -b 4096 -C "hexo-deploy" -f hexo-deploy-key
# 会生成两个文件：
# hexo-deploy-key     （私钥）
# hexo-deploy-key.pub （公钥）

步骤2：在部署仓库添加公钥

  • 打开 yourname.github.io 仓库
  • Settings → Deploy keys → Add deploy key
  • Title: HEXO_DEPLOY
  • Key: 粘贴 hexo-deploy-key.pub 的内容
  • ✅ 勾选 “Allow write access”

步骤3：在源码仓库添加私钥

  • 打开 hexo-source 仓库
  • Settings → Secrets and variables → Actions → New repository secret
  • Name: HEXO_DEPLOY_KEY
  • Secret: 粘贴 hexo-deploy-key 的内容（私钥）

4. 添加协作者

  • 打开 hexo-source 仓库
  • Settings → Collaborators → Add people
  • 输入协作者 GitHub 用户名

6.3 协作者工作流程

# 1. 克隆源码仓库（不是部署仓库！）
git clone https://github.com/yourname/hexo-source.git
cd hexo-source

# 2. 安装依赖
npm install

# 3. 创建新分支
git checkout -b article/new-post

# 4. 写作
hexo new "新文章标题"
# 编辑文章...

# 5. 本地预览
hexo clean && hexo g && hexo s

# 6. 提交代码
git add .
git commit -m "add: 新文章《标题》"
git push origin article/new-post

# 7. 创建 Pull Request
# 在 GitHub 上合并到 main 分支

# 8. 自动部署
# GitHub Actions 会自动构建并部署到 yourname.github.io

⚠️ 重要提醒：

  • 协作者只需要克隆 hexo-source 仓库
  • 不要直接修改 yourname.github.io 仓库（会被自动覆盖）
  • 所有写作都在 hexo-source 仓库完成

6.4 GitHub Actions 自动部署（双仓库）

在 hexo-source 仓库创建 .github/workflows/deploy.yml：

name: Deploy Hexo Blog

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      # 1. 检出源码
      - uses: actions/checkout@v3
        with:
          submodules: true
          fetch-depth: 0

      # 2. 设置 Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'

      # 3. 安装依赖
      - name: Install Dependencies
        run: npm install

      # 4. 构建（PR时不部署）
      - name: Build
        run: hexo generate

      # 5. 部署到 yourname.github.io（仅 main 分支推送时）
      - name: Deploy
        if: github.event_name == 'push' && github.ref == 'refs/heads/main'
        uses: peaceiris/actions-gh-pages@v3
        with:
          deploy_key: ${{ secrets.HEXO_DEPLOY_KEY }}
          external_repository: yourname/yourname.github.io
          publish_dir: ./public
          publish_branch: main
          commit_message: "deploy: ${{ github.event.head_commit.message }}"

配置说明：

配置项 说明
deploy_key 使用 Deploy Key 方式推送（更安全）
external_repository 部署目标仓库（yourname.github.io）
publish_dir 构建输出目录（Hexo 默认是 public）
publish_branch 目标仓库的分支（main）

工作流程：

  1. 推送代码到 hexo-source 的 main 分支
  2. GitHub Actions 自动触发构建
  3. 生成静态文件到 public 目录
  4. 使用 Deploy Key 推送到 yourname.github.io
  5. GitHub Pages 自动更新网站

6.6 图片管理方案

方案 1：本地图片（适合少量图片）

source/_posts/文章名/
├── index.md
└── images/
    └── screenshot.png

引用方式：

![描述](./images/screenshot.png)

缺点：

  • 仓库体积会变大
  • 多人协作时图片冲突
  • 不适合大量图片

方案 2：GitHub + jsDelivr（推荐）

利用 GitHub 仓库作为图床，jsDelivr CDN 加速：

步骤：

  1. 创建图床仓库

    • 新建仓库：blog-images
    • 设置为公开

  2. 上传图片

    # 克隆图床仓库
git clone https://github.com/yourname/blog-images.git
cd blog-images

# 创建目录结构
mkdir -p 2026/03

# 复制图片
cp ~/Downloads/screenshot.png 2026/03/

# 提交
git add .
git commit -m "add: 添加截图"
git push

  3. 获取图片链接

    https://cdn.jsdelivr.net/gh/yourname/blog-images@main/2026/03/screenshot.png

  4. 在文章中使用

    ![描述](https://cdn.jsdelivr.net/gh/yourname/blog-images@main/2026/03/screenshot.png)

优点：

  • ✅ 免费
  • ✅ CDN 加速（全球访问快）
  • ✅ 版本控制
  • ✅ 不增加博客仓库体积

方案 3：商业图床

图床 免费额度 特点
阿里云 OSS 5GB/月 稳定可靠
腾讯云 COS 50GB/月 国内速度快
七牛云 10GB/月 有免费域名
SM.MS 5GB 简单易用

推荐： 个人博客用 GitHub + jsDelivr 足够，企业项目用阿里云/腾讯云。

图片压缩建议

上传前压缩图片，减少加载时间：

  • TinyPNG - 在线压缩
  • ImageOptim - Mac 工具
  • Squoosh - Google 出品

七、多设备协作写作

很多小伙伴在公司用 PC，回家用自己的笔记本，如何在多台设备上无缝协作写作？

7.1 方案对比

方案 优点 缺点 推荐度
Git 同步（推荐） 版本控制、历史记录、多人协作 需要学习 Git ⭐⭐⭐⭐⭐
云盘同步（OneDrive/坚果云） 简单、自动同步 冲突处理麻烦、无版本控制 ⭐⭐⭐
U盘拷贝 简单直接 容易丢失、版本混乱 ⭐

7.2 Git 多设备方案（推荐）

核心思路： 用 GitHub 作为中央仓库，各设备通过 Git 同步。

设备A（公司 PC）初始化

# 1. 克隆源码仓库（第一次）
git clone https://github.com/yourname/hexo-source.git
cd hexo-source

# 2. 安装依赖
npm install

# 3. 日常写作
hexo new "新文章"
# 编辑文章...

# 4. 提交代码
hexo clean && hexo g && hexo s  # 本地预览
git add .
git commit -m "add: 新文章《标题》"
git push origin main

设备B（家里笔记本）同步

# 1. 克隆源码仓库（第一次）
git clone https://github.com/yourname/hexo-source.git
cd hexo-source
npm install

# 2. 每天开始写作前 - 拉取最新代码
git pull origin main

# 3. 继续写作...
hexo new "另一篇文章"
# 编辑...

# 4. 提交
git add .
git commit -m "add: 另一篇文章"
git push origin main

回到设备A（第二天）

# 拉取家里写的文章
git pull origin main

# 继续写作...

7.3 多设备工作流程

公司 PC                    GitHub 仓库                家里笔记本
   |                            |                            |
   ├── 写作文章A ──────────────┼────────────────────────────┤
   │   git push ───────────────┼────────────────────────────┤
   │                           │◄────────────────────────────┤
   │                           │                            ├── git pull
   │                           │                            │   获取文章A
   │                           │                            ├── 继续写作B
   │                           │◄────────────────────────────┤
   │                           │                            │   git push
   ├── git pull ───────────────┤                            │
   │   获取文章B ──────────────┤                            │
   │   继续写作C ──────────────┤                            │
   │   git push ───────────────┤                            │
   │                           │                            ├── git pull
   │                           │                            │   获取文章C

7.4 关键原则

原则1：写作前总是先 pull

# 每天开始写作前
git pull origin main

# 然后再写作
hexo new "文章"
# 编辑...

原则2：写作后及时 push

# 完成一个阶段（比如写完一节）就提交
git add .
git commit -m "add: 完成文章第一节"
git push origin main

原则3：不要长时间不提交

❌ 错误：写了一整天，回家才提交 → 容易冲突
✅ 正确：写完一节就提交 → 减少冲突

7.5 冲突处理（多设备）

如果两台设备都修改了同一篇文章：

# 设备A 提交并推送
git add .
git commit -m "update: 修改文章A"
git push origin main

# 设备B 尝试推送（会失败）
git add .
git commit -m "update: 也修改了文章A"
git push origin main
# ❌ 错误：rejected

# 设备B 需要先拉取合并
git pull origin main
# 出现冲突标记

# 编辑冲突文件，解决冲突
# <<<<<<< HEAD
# 设备B的修改
# =======
# 设备A的修改
# >>>>>>> main

# 保留需要的版本，删除标记

# 重新提交
git add .
git commit -m "merge: 合并设备A和B的修改"
git push origin main

7.6 实用技巧

技巧1：查看修改内容

# 查看哪些文件被修改
git status

# 查看具体修改内容
git diff

# 查看某篇文章的修改
git diff source/_posts/文章名.md

技巧2：临时保存修改（stash）

# 临时保存当前修改（不想提交，但要切换分支）
git stash

# 恢复修改
git stash pop

技巧3：查看提交历史

# 查看最近提交
git log --oneline -10

# 查看某篇文章的修改历史
git log --oneline source/_posts/文章名.md

技巧4：撤销误操作

# 撤销最后一次提交（保留修改）
git reset --soft HEAD~1

# 撤销对某篇文章的修改（恢复到上次提交）
git checkout -- source/_posts/文章名.md

7.7 配置 SSH 免密（每台设备）

避免每次 push/pull 都输入密码：

# 生成 SSH 密钥（每台设备做一次）
ssh-keygen -t ed25519 -C "公司PC" -f ~/.ssh/id_ed25519_work
ssh-keygen -t ed25519 -C "家里笔记本" -f ~/.ssh/id_ed25519_home

# 添加公钥到 GitHub
# Settings -> SSH and GPG keys -> New SSH key

# 测试连接
ssh -T git@github.com

7.8 总结

多设备协作的核心：

  1. ✅ 所有设备都从 GitHub 克隆同一个仓库
  2. ✅ 写作前 git pull 获取最新内容
  3. ✅ 写作后 git push 提交到云端
  4. ✅ 及时提交，不要积压太多修改
  5. ✅ 冲突时手动合并，保留需要的版本

这样，公司写一半，回家继续写，无缝衔接！
