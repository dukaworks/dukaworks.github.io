告别浏览器书签！我用一天，从零撸一个高颜值 Home-Lab 导航页

发表于 2026-03-07 分类于 技术分享 阅读次数： 本文字数： 1.8k 阅读时长 ≈ 7 分钟

先上最终成品图，这颜值你打几分？

是不是感觉还不错？最关键的是，它最终的交付物只有一个 HTML 文件，无需任何后端，无需数据库，甚至无需 Web 服务器（直接用浏览器打开就行），真正做到了“开箱即用”！

市面上有很多优秀的导航页项目，比如 Homer、Heimdall 等，但它们要么需要复杂的配置，要么界面风格不是我的菜。我想要的是一个 极致简约、颜值在线、高度定制、而且完全由我掌控 的东西。

于是，一个念头涌上心头：何不自己动手，丰衣足食？

今天，我就带大家复盘一下，我是如何用一天时间，从一个空白的 index.html 文件开始，最终打造出一个让我自己都惊艳的、功能完备的个人导航仪表盘。

第一阶段：奠定骨架与颜值 (HTML & CSS)

一个项目的颜值决定了我们打开它的欲望。所以，我先把重心放在了 UI/UX 上。

1. 审美方向：

我想要的风格是**“科技感”和“精致感”**。最终确定了几个核心设计元素：

  • 深色模式优先 (Dark Mode First)：对我们这些 7x24 小时盯着屏幕的 “LSP” 们更友好。
  • 玻璃拟态 (Glassmorphism)：通过 backdrop-filter: blur() 实现毛玻璃效果，让卡片背景半透明，透出背景的星云光斑，层次感拉满。
  • 辉光效果 (Glow Effect)：鼠标悬停在卡片上时，卡片轻微上浮，并伴有与其主色调一致的柔和辉光，提供流畅的交互反馈。
  • CSS 变量：这是整个项目能够轻松定制和维护的关键。我把所有的主题颜色都定义成了 CSS 变量，未来想做“一键换肤”功能？分分钟的事！
/* 设计灵魂：CSS 颜色变量 */
:root {
    --bg-color: #171E2B;
    --card-bg-color: rgba(30, 41, 59, 0.6);
    --text-color: #E2E8F0;
    --glow-color: rgba(88, 166, 255, 0.4);
    --accent-color: #58A6FF;
}

/* 玻璃拟态和辉光效果的核心代码 */
.service-card {
    background-color: var(--card-bg-color);
    backdrop-filter: blur(10px);
    transition: transform 0.3s ease, box-shadow 0.3s ease;
}

.service-card:hover {
    transform: translateY(-8px);
    box-shadow: 0 0 25px 0 var(--glow-color);
}

2. 页面布局：

采用了经典的响应式网格布局 (display: grid)，让卡片能根据屏幕宽度自动排列，PC、平板、手机都能完美适配。一个悬浮操作按钮 (FAB) 固定在右下角，用于添加新服务，交互逻辑清晰。

第二阶段：注入灵魂 (JavaScript)

光有漂亮的壳子不行，得让它“活”起来。JS 的部分，我遵循了**“数据驱动视图”**的现代前端思想。

1. 数据的增删改查 (CRUD) 与持久化

这是整个应用的逻辑核心。我需要能够自由地添加、编辑和删除我的服务卡片。

数据存在哪？
最开始我想用一个外部的 services.json 文件，但很快就遇到了天坑：浏览器安全策略禁止本地 HTML 文件通过 fetch 读取本地其他文件。这意味着不启动一个 Web Server 就无法使用。这违背了我的“极致简约”初衷。

最终方案：

  • 默认数据：直接写成一个 JS 对象数组，内置在 <script> 标签里。首次打开就有内容，完美解决了本地加载问题。
  • 用户数据：用户的任何修改（增、删、改）全部通过 localStorage API 存储在浏览器本地。这意味着你的定制化配置会跟着浏览器走，只要不清理缓存，数据就永远都在。完美实现了无后端的持久化！
// 数据驱动的核心：加载数据 -> 渲染页面
function loadData() {
    const storedData = localStorage.getItem('dashboard_services');
    dashboardData = storedData ? JSON.parse(storedData) : getDefaultData();
    renderPage(); // 根据数据重新绘制整个页面
}

function saveData() {
    localStorage.setItem('dashboard_services', JSON.stringify(dashboardData));
}

2. 全场最佳：纯前端图片上传

这是我最想“吹”的一个功能点。通常，一提到“图片上传”，大家想到的就是 FormData、后端接收、存储为文件、返回 URL… 一套复杂的流程。

但对于我们这种小图标的需求，完全可以用更“黑科技”的纯前端方案解决：FileReader API + Base64 Data URL。

它的原理就像魔法：

  1. 用户选择一张本地图片。
  2. JS 的 FileReader 对象在浏览器里直接读取图片文件的二进制内容。
  3. 然后将其编码成一长串文本字符（Base64）。
  4. 这串文本本身就可以像一个 URL 一样直接用在 <img> 标签的 src 属性里！
// “黑科技”核心代码
function handleIconUpload(event) {
    const file = event.target.files[0];
    const reader = new FileReader();
    reader.onload = (e) => {
        // e.target.result 就是那串神奇的 Base64 文本
        iconUrlInput.value = e.target.result;
    };
    reader.readAsDataURL(file);
}

最终，这串 Base64 文本和服务的标题、描述等信息一起，被当做普通文本存进了 localStorage。这意味着，你的图标也被“存”进了浏览器里，实现了图标的永久离线访问！

第三阶段：部署上线 (Docker & Nginx Proxy Manager)

虽然这个 index.html 在本地就能用，但作为一个有追求的 Home-Lab 玩家，必须给它一个域名，让它能被优雅地访问。

这里我用了我的“三驾马车”：Docker + Nginx + Nginx Proxy Manager (NPM)。

  1. 打包成 Docker 镜像：我用 nginx:alpine 这个超轻量级的镜像作为 Web 服务器，只需要把我们的 index.html 文件挂载到它的网站根目录就行。

  2. docker-compose 管理：为了方便管理，我把 docker run 命令转换成了 docker-compose.yml 文件。最关键的一步是，让这个新的 Nginx 容器加入到 NPM 已经创建的代理网络中。

    version: '3.8'
services:
  my-website:
    image: nginx:alpine
    container_name: my-website
    volumes:
      - /path/to/your/html_folder:/usr/share/nginx/html:ro # 把存放 index.html 的目录挂载进去
    networks:
      - proxy-net # 连接到外部网络
    restart: unless-stopped
networks:
  proxy-net:
    name: nginx-proxy-manager_proxy-net # 这里填你NPM所在的真实网络名
    external: true # 声明这是个已存在的外部网络

  3. NPM 配置反向代理：最后在 NPM 里，添加一个新的代理主机，域名填你准备好的，转发主机名直接填我们容器的名字 my-website，端口 80。再申请一个 SSL 证书，开启 HTTPS。

总结与源码

这次从零到一的经历，让我再次感受到了创造的乐趣。我们没有依赖任何复杂的前端框架（React/Vue），而是回归本源，用最基础的 HTML、CSS 和原生 JS，构建了一个高度实用且极具美感的作品。

它证明了，只要思路清晰，掌握核心技术，我们就能打造出真正属于自己的、贴合需求的工具。

未来的计划？
这个 v0.1.0 版本已经很完美了，但折腾永无止境！后续我可能会给它加上：

  • 拖拽排序：自由调整卡片和分类的顺序。
  • 主题切换：一键切换浅色模式或更多自定义主题。
  • 服务状态检测：给状态小圆点加上真实的网络请求，实时显示服务的在线状态。
