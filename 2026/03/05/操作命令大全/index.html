Docker 操作命令大全

发表于 2026-03-05 分类于 学习探索 阅读次数： 本文字数： 2k 阅读时长 ≈ 7 分钟

当然。作为一名资深的软件工程师，Docker 是我们工具箱中不可或缺的核心组件。它彻底改变了我们构建、分发和运行应用的方式。

下面我为你整理了一份详尽的 Docker 命令行操作大全。同样地，我将按照实际工作流和使用场景来组织，并附上高级工程师的实践建议和注意事项，确保这份大全不仅全面，而且实用。

Docker 命令行操作大全 (The Senior Engineer’s Docker Cheatsheet)

在开始之前，先明确两个核心概念：

  • 镜像 (Image): 一个只读的模板，包含了运行应用程序所需的文件系统和配置。可以把它理解为软件的“安装包”或面向对象编程中的“类”。
  • 容器 (Container): 镜像的运行实例。它拥有自己独立的文件系统、网络和进程空间。可以把它理解为软件的“运行程序”或“对象实例”。

一、 镜像管理 (Image Management)

一切从镜像开始。

# 从 Docker Hub 或其他镜像仓库拉取一个镜像
# 格式: docker pull <repository>:<tag>
docker pull nginx:latest
docker pull ubuntu:20.04

# 列出本地已有的所有镜像
docker images
# 或者
docker image ls

# 删除一个或多个本地镜像
# 可以使用镜像名:标签 或 镜像ID
docker rmi nginx:latest
docker rmi <image_id_1> <image_id_2>

# 强制删除一个镜像 (即使有容器正在使用它)
docker rmi -f <image_id>

# 为镜像打上新的标签 (常用于推送前)
# 格式: docker tag <source_image>:<tag> <target_image>:<tag>
docker tag nginx:latest my-registry.com/my-app/nginx:1.0

# 登录到 Docker 镜像仓库
docker login <registry_url>  # 默认为 Docker Hub

# 将本地镜像推送到远程仓库 (必须先登录和打好标签)
docker push my-registry.com/my-app/nginx:1.0

# 查看镜像的构建历史 (有助于理解镜像的分层结构)
docker history <image_id>

✨ 资深工程师建议： 始终为你的镜像指定明确的 tag（如 ubuntu:20.04），避免使用隐式的 :latest。这能保证你的构建是可复现的，防止因上游镜像更新导致意外的 break。

二、 容器生命周期管理 (Container Lifecycle Management)

这是 Docker 最核心、最常用的部分。

# 1. 创建并运行一个容器
# 格式: docker run [OPTIONS] IMAGE [COMMAND] [ARG...]
# -d: 后台运行 (detached mode)
# -p: 端口映射 (格式: <host_port>:<container_port>)
# --name: 给容器指定一个名字
# --rm: 容器停止后自动删除
docker run -d -p 8080:80 --name my-nginx-container nginx

# 列出当前正在运行的容器
docker ps
# 或者
docker container ls

# 列出所有容器 (包括已停止的)
docker ps -a

# 停止一个正在运行的容器
docker stop <container_id_or_name>

# 启动一个已经停止的容器
docker start <container_id_or_name>

# 重启一个容器
docker restart <container_id_or_name>

# 删除一个或多个已停止的容器
docker rm <container_id_1> <container_id_2>

# 强制删除一个正在运行的容器 (不推荐，应先 stop)
docker rm -f <container_id>

# 暂停容器中的所有进程
docker pause <container_id>
# 恢复容器中的所有进程
docker unpause <container_id>

三、 与容器交互 (Interacting with Containers)

容器运行起来后，你需要查看它的状态和内部情况。

# 查看容器的日志输出
docker logs <container_id_or_name>

# 实时跟踪容器的日志输出 (类似 tail -f)
docker logs -f <container_id_or_name>

# 在一个正在运行的容器中执行一个命令
# 这是进入容器内部进行调试的常用方法
# -i: 交互模式 (interactive)
# -t: 分配一个伪终端 (tty)
docker exec -it <container_id_or_name> /bin/bash
# 注意: 对于 alpine 等极简镜像，可能没有 bash，应使用 sh
# docker exec -it <container_id_or_name> /bin/sh

# 查看容器的详细信息 (包括 IP 地址、挂载点等)
docker inspect <container_id_or_name>

# 查看容器内运行的进程
docker top <container_id_or_name>

# 从主机复制文件到容器
docker cp ./local-file.txt <container_id>:/path/in/container/

# 从容器复制文件到主机
docker cp <container_id>:/path/in/container/remote-file.txt ./

✨ 资深工程师建议： 优先使用 docker logs 进行调试。尽量避免频繁 docker exec 进入容器修改内容，这违背了容器“不可变基础设施”的理念。需要修改时，应更新镜像并重新创建容器。

四、 构建镜像 (Building Images with Dockerfile)

将你的应用打包成标准化的镜像。

# 在包含 Dockerfile 的目录下构建镜像
# -t: 指定镜像的名称和标签 (tag)
# . : 代表 Dockerfile 的上下文路径 (context path) 为当前目录
docker build -t my-app:1.0 .

# 构建时指定 Dockerfile 文件名
docker build -t my-app:1.0 -f MyDockerfile .

# 构建时不使用缓存 (用于确保最新)
docker build --no-cache -t my-app:1.0 .

一个简单的 Dockerfile 示例：

# 使用一个基础镜像
FROM python:3.9-slim

# 设置工作目录
WORKDIR /app

# 复制依赖文件并安装
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 复制应用代码
COPY . .

# 暴露端口
EXPOSE 5000

# 容器启动时执行的命令
CMD ["python", "app.py"]

五、 数据管理 (Data Management with Volumes)

让数据在容器之外持久化，或者在多个容器间共享。

# 创建一个 Docker 管理的卷 (volume)
docker volume create my-data-volume

# 列出所有卷
docker volume ls

# 查看卷的详细信息
docker volume inspect my-data-volume

# 删除一个卷
docker volume rm my-data-volume

# 运行容器时挂载一个命名的卷
# -v <volume_name>:<path_in_container>
docker run -d -v my-data-volume:/var/lib/mysql mysql

# 运行容器时挂载一个主机目录 (bind mount)
# -v <host_path>:<container_path>
docker run -d -v /path/on/host:/app/data my-app:1.0

✨ 资深工程师建议： 优先使用命名的卷 (-v my-data-volume:/data) 而不是绑定挂载主机目录 (-v /host/path:/data)。前者由 Docker 管理，更具可移植性，且性能通常更好。

六、 网络管理 (Network Management)

让容器之间、容器与外界之间能够通信。

# 创建一个自定义的桥接网络
# 自定义网络内的容器可以通过容器名相互访问
docker network create my-app-network

# 运行容器并将其连接到自定义网络
docker run -d --name db --network my-app-network mysql
docker run -d --name api --network my-app-network -p 8000:8000 my-api

# 在 api 容器内部，可以直接通过 `ping db` 来访问数据库容器

# 列出所有网络
docker network ls

# 查看网络的详细信息
docker network inspect my-app-network

# 断开/连接容器与网络的连接
docker network disconnect my-app-network db
docker network connect my-app-network db

七、 系统清理 (System & Cleanup)

Docker 使用时间长了会产生很多无用的镜像、容器和卷，需要定期清理。

# 一键清理！删除所有已停止的容器、未被任何容器使用的网络、悬空的镜像 (dangling images) 和构建缓存
docker system prune

# 更彻底的清理：在上面命令的基础上，还会删除所有未被至少一个容器使用的镜像
docker system prune -a

# 单独删除所有已停止的容器
docker container prune
# 快捷命令: docker rm $(docker ps -aq)

# 单独删除所有悬空的镜像 (没有标签的镜像层)
docker image prune
# 快捷命令: docker rmi $(docker images -f "dangling=true" -q)

# 单独删除所有未被使用的卷
docker volume prune

⚠️ 重要警告： prune 命令非常方便，但也可能删除你不想删除的东西。在生产环境执行前请务必确认。

下一步：Docker Compose

当你的应用包含多个容器时（例如一个 Web 服务 + 一个数据库 + 一个缓存），手动管理 docker run 命令会变得非常繁琐。这时就应该使用 Docker Compose。

Docker Compose 允许你使用一个 docker-compose.yml 文件来定义和运行一个多容器的 Docker 应用。

一个简单的 docker-compose.yml 文件示例：

version: '3.8'

services:
  web:
    build: .
    ports:
      - "8000:5000"
    volumes:
      - .:/code
    depends_on:
      - redis

  redis:
    image: "redis:alpine"

常用命令：

# 在后台启动并构建所有服务
docker-compose up -d --build

# 停止并删除所有服务相关的容器、网络
docker-compose down

# 查看所有服务的日志
docker-compose logs -f

这份大全覆盖了 Docker 日常开发和运维中的绝大多数场景。精通这些命令，你就能高效地驾驭容器化技术。祝你在容器化的世界里游刃有余！
